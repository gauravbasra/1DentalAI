# Payer Matrix Schema And Service Slice

Status: Backend schema/service slice implemented  
Date: 2026-05-23  
Owner: Payer Integration Worker

## Purpose

Define the first Stedi-style payer matrix slice for 1DentalAI. The public Stedi network page presents payer name/ID rows with transaction columns for eligibility checks, COB, claim submission, claim acknowledgment, ERA, attachments, and real-time claim status. This artifact maps that shape into PMS-grade backend objects, then extends it for dental-specific prior authorization, portal/RPA fallback, credential vault references, run logs, generated PDF/EOB artifacts, enrollment readiness, and a production gate for payer-facing actions.

## Scope

This slice covers payer discovery, route readiness, credential-reference inventory, RPA run logging, and artifact references. It does not submit eligibility, claims, attachments, status checks, prior authorizations, or ERA enrollment yet.

First supported transaction families:

- `ELIGIBILITY_270_271`
- `COB_271`
- `CLAIM_837D`
- `CLAIM_ACK_277CA`
- `CLAIM_STATUS_276_277`
- `ERA_835`
- `ATTACHMENT_275`
- `PRIOR_AUTH`

## Schema Direction

| Model | Purpose | Required fields |
| --- | --- | --- |
| `PayerRegistryEntry` | Canonical payer/trading-partner record used by insurance, claims, and RCM. | `id`, `payerName`, `primaryPayerId`, `payerType`, `active`, `source`, `sourceUpdatedAt` |
| `PayerAlias` | Search aliases and normalized names from cards, PMS plans, and clearinghouse directories. | `payerRegistryEntryId`, `alias`, `aliasType`, `confidence` |
| `PayerTransactionCapability` | Matrix row for transaction support by payer and service line. Stores Stedi-style transaction identity plus clearinghouse payer IDs and enrollment/fallback requirements. | `payerRegistryEntryId`, `transactionFamily`, `x12Transaction`, `stediTransactionKey`, `supported`, `clearinghouse`, `clearinghousePayerId`, `payerEnrollmentMode`, `requiresEnrollment`, `requiresProviderEnrollment`, `requiresLocationEnrollment`, `requiresPayerPortalFallback`, `portalFallbackReason`, `lastVerifiedAt` |
| `PayerNetworkModel` | Production gate model for commercial, government, direct payer, clearinghouse, and portal-only routing. Tracks credentialing state independently from payer enrollment. | `payerRegistryEntryId`, `networkType`, `routeType`, `clearinghouse`, `clearinghouseRouteKey`, `clearinghousePayerId`, `directEndpoint`, `portalInstructions`, `portalRpaProfile`, `manualFallbackAllowed`, `enrollmentStatus`, `credentialingStatus`, `credentialingOwnerRole`, `credentialingDueAt`, `effectiveFrom`, `effectiveTo` |
| `PayerRoutePolicy` | Runtime decision record used before any payer-facing action. Captures clearinghouse route decision metadata and credential/connector requirements. | `payerRegistryEntryId`, `transactionFamily`, `preferredRouteType`, `fallbackRouteType`, `approvalPolicy`, `proofRequired`, `routeDecisionMetadata`, `externalActionBlockedReason`, `requiresValidatedCredential`, `connectorHealthRequired`, `clearinghouseRouteDecision` |
| `PayerMatrixSnapshot` | Immutable imported matrix version for audit and rollback. | `id`, `source`, `version`, `importedAt`, `rowCount`, `checksum`, `importedByUserId` |
| `PayerPortalCredentialReference` | Metadata-only link between a payer portal and the encrypted `ConnectorCredentialVault`. It never stores portal passwords or API secrets. | `payerRegistryEntryId`, `portalHost`, `credentialVaultId`, `credentialStatus`, `credentialScope`, `ownerRoleKey`, `lastValidatedAt`, `rotationDueAt` |
| `PayerRpaRunLog` | Immutable-ish run record for screen-scraping bots that log into payer portals for eligibility, EOB, prior auth, attachment, or claim status work. | `payerRegistryEntryId`, `credentialReferenceId`, `transactionFamily`, `sourceObjectType`, `sourceObjectId`, `runStatus`, `botName`, `portalHost`, `startedAt`, `completedAt`, `evidenceUri`, `errorMessageRedacted`, `noPhiInLogs` |
| `PayerGeneratedArtifactReference` | Reference to PDFs, EOB packets, ERA posting reports, prior auth packets, eligibility PDFs, screenshots, and attachment packets generated by payer integrations/RPA. | `payerRegistryEntryId`, `rpaRunLogId`, `sourceObjectType`, `sourceObjectId`, `artifactType`, `title`, `storageUri`, `checksum`, `documentStatus`, `reviewedByRole` |

Enums should be narrow and auditable:

- `PayerType`: `COMMERCIAL`, `MEDICAID`, `MEDICARE_ADVANTAGE`, `CHIP`, `DISCOUNT_PLAN`, `UNKNOWN`
- `PayerRouteType`: `CLEARINGHOUSE`, `DIRECT_PAYER`, `PAYER_PORTAL`, `MANUAL_ONLY`, `BLOCKED`
- `PayerNetworkType`: `IN_NETWORK`, `OUT_OF_NETWORK`, `PAR_NEGOTIATED`, `GOVERNMENT_PROGRAM`, `UNKNOWN`
- `PayerApprovalPolicy`: `AUTO_INTERNAL_ONLY`, `HUMAN_APPROVAL_REQUIRED`, `CONNECTOR_ACK_REQUIRED`, `MANUAL_PROOF_REQUIRED`, `BLOCKED`

## Service Slice

The backend service in `src/lib/payer-network-repository.ts` implements these functions:

| Function | Contract |
| --- | --- |
| `searchPayerMatrix(query)` | Returns canonical payer matches plus aliases and confidence. |
| `getPayerRouteReadiness(payerId, transactionFamily, context)` | Returns supported capability, route policy, enrollment gaps, provider/location blockers, and required proof. |
| `importPayerMatrixSnapshot(snapshot)` | Imports a versioned matrix and writes immutable snapshot metadata before updating active rows. |
| `resolveInsurancePlanPayer(plan)` | Maps PMS insurance plan/card data to a `PayerRegistryEntry` with confidence and unresolved reasons. |
| `assertPayerProductionGate(input)` | Blocks payer-facing actions unless network model, route policy, enrollment readiness, connector health, and approval/proof rules pass. |
| `registerPortalCredentialReference(input)` | Stores only vault reference metadata for payer portals and RPA bots. Plaintext secrets stay in `ConnectorCredentialVault`. |
| `createPayerRpaRunLog(input)` | Starts a payer portal/RPA run only when route policy permits portal automation and a credential reference is validated. |
| `recordPayerGeneratedArtifact(input)` | Registers generated eligibility PDFs, prior auth PDFs, EOB PDFs, ERA posting PDFs, claim attachment packets, and payer screenshot references. |
| `getPayerMatrixCoverage(tenantId)` | Aggregates matrix coverage by transaction family, supported payer count, portal fallback count, and enrollment-required count. |

### Stedi-Style Transaction Mapping

The repository includes a concrete `STEDI_STYLE_TRANSACTION_REQUIREMENTS` map rather than only storing display tokens. Each transaction family carries:

| Family | X12 transaction | Stedi-style key | Proof/readiness examples |
| --- | --- | --- | --- |
| `ELIGIBILITY_270_271` | `270/271` | `healthcare.eligibility` | 271 response, eligibility PDF, payer portal screenshot/reference |
| `COB_271` | `271 COB` | `healthcare.cob.eligibility` | 271 COB segment, payer portal screenshot/reference |
| `CLAIM_837D` | `837D` | `healthcare.claim.dental` | clearinghouse trace, 277CA acknowledgement, submitted claim packet |
| `CLAIM_ACK_277CA` | `277CA` | `healthcare.claim.acknowledgement` | 277CA acknowledgement, clearinghouse acknowledgement ID |
| `CLAIM_STATUS_276_277` | `276/277` | `healthcare.claim_status` | 277 claim status response, payer portal screenshot/reference |
| `ERA_835` | `835` | `healthcare.payment.era` | 835 ERA, EOB PDF, ERA posting PDF |
| `ATTACHMENT_275` | `275` | `healthcare.claim_attachment` | 275 attachment acknowledgement, claim attachment packet |
| `PRIOR_AUTH` | `278` | `healthcare.prior_authorization` | 278 response, prior auth PDF, payer portal screenshot/reference |

`importPayerMatrixSnapshot` now rejects sparse/token-only rows before write: a row must include payer identity, at least three concrete transaction capabilities, a network route, and matching route policies. Portal/RPA fallback rows must include a portal URL. Supported capabilities require a preferred route type.

### Clearinghouse Route Decision Metadata

Route policy metadata must identify enough information for a connector worker or RCM reviewer to understand the decision:

- `clearinghouse`
- `clearinghouseRouteKey`
- `clearinghousePayerId`
- `stediTransactionKey`
- `routePurpose`
- `clearinghouseRouteDecision`
- `connectorHealthRequired`
- `requiresValidatedCredential`

Readiness fails closed when a clearinghouse route lacks a payer ID, when the clearinghouse decision is not `PREFERRED`, `APPROVED`, or `AVAILABLE`, when connector health metadata lacks the Stedi transaction key, or when credentialing is `PENDING`, `BLOCKED`, `EXPIRED`, or `DENIED`.

### Eligibility Portal RPA Evidence Contract

The eligibility portal browser runner returns both raw scrape values for in-process normalization and a PHI-safe `browserEvidenceManifest` for persistence. The manifest links the run to artifact preparation and screen-scrape storage through `sourceTraceId`, screenshot checksum, `fieldEvidenceHash`, selector evidence fingerprints, missing selector keys, low-confidence keys, DOM `extractionAttempts`, reusable `sessionState`, and `layoutDriftEvidence`. It must assert `noPhiInLogs`.

Artifact preparation stores the manifest summary on the screenshot reference and eligibility PDF metadata. Screen-scrape recording rejects a manifest when its screenshot checksum does not match the screenshot artifact, when a provided `sourceTraceId` differs, or when selector evidence is missing for scraped fields. Persisted `rawFieldEvidence` intentionally excludes scraped text and normalized values; it keeps selector, confidence, region, and fingerprint only.

Payer portal runs can reuse Playwright storage state when a validated session exists, then save refreshed session state after a successful run. `manualCheckpoint` and configured MFA selectors route the run to `NEEDS_MANUAL_REVIEW` with a PHI-safe error code rather than pretending the portal was verified. Retry taxonomy is explicit: host escape is `BLOCKED`, MFA/checkpoint is manual review, timeout-style failures are retryable, and authentication failures are not retried. PMS writeback is staged as a sectioned payload for `PmsPatientInsurance`, `PmsBenefitSummary`, and audit metadata so reviewers can see exactly which PMS sections will change.

Initial consumers:

- PMS insurance page: payer search, unresolved payer warnings, eligibility readiness.
- RCM queue: claim route readiness, attachment/claim-status fallback.
- Credentialing workbench: provider/entity/location enrollment gaps by payer.
- Connector setup: matrix import, connector smoke-test status, route availability.

## Production Gate Expectation

Any service that marks a payer-facing action as submitted, verified, acknowledged, paid, or enrolled must first call `assertPayerProductionGate`.

The gate must fail closed when any of these are missing:

- Active `PayerRegistryEntry`.
- Active `PayerNetworkModel` for the payer and transaction context.
- Supported `PayerTransactionCapability` for the transaction family.
- Current provider/entity/location enrollment when required.
- Current credentialing status for provider/entity/location payer participation.
- Healthy connector route or explicit manual proof workflow.
- Human approval when the route policy requires it.
- Immutable audit event with request source, actor, route decision, and proof reference.
- Validated `PayerPortalCredentialReference` when `allowPortalRpa` or portal/manual fallbacks are required.
- `PayerRpaRunLog` and `PayerGeneratedArtifactReference` when bots read payer portal screens, generate EOBs, or create a generated PDF artifact used by staff.
- Clearinghouse route decision metadata with payer ID and Stedi-style transaction key for electronic routes.

The gate must not allow `PayerNetworkType.UNKNOWN`, `PayerRouteType.BLOCKED`, stale matrix snapshots, or portal/manual fallbacks to be treated as electronic acknowledgement.

## Implemented Backend Artifacts

- `prisma/schema.prisma` now defines the payer registry, aliases, transaction capabilities, network models, route policies, matrix snapshots, RPA credential references, run logs, and generated artifact references.
- `prisma/migrations/202605230900_payer_network_matrix/migration.sql` creates the same backend tables and indexes.

It also rejects token-only Stedi mapping stubs by requiring per-family `x12Transaction`, `stediTransactionKey`, and proof mapping.
